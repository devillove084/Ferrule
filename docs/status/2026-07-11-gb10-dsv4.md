## Correctness state

### Closed numerical problems

1. **Missing non-compressed decode KV QAT**

   Both CUDA non-compressed decode paths now apply attention KV QAT after KV
   RoPE, matching batched prefill and reference semantics.

2. **Non-deterministic MoE accumulation**

   Routed experts write to per-expert scratch. A route-ranked device reducer starts
   from the shared-FFN output and folds expert contributions in router-rank order.
   The production path no longer relies on relaxed atomic accumulation or a
   differently parenthesized reduction.

3. **Compressed decode numerical mismatch**

   Incremental compressor output is determined by the same CUDA softmax, RMS,
   RoPE, and QAT chain as batched prefill. Host shadow state remains for control
   bookkeeping but no longer defines compressed values.

4. **Combined KV capacity growth**

   Growth preserves authoritative device window/compressed values using D2D copy.
   Replacement is installed only after allocation/copy succeeds. Redundant
   post-host-commit growth was removed.

5. **RoPE fixed capacity**

   RoPE tables bind dimension, complete YaRN parameters, and capacity. They retain
   a 4096 initial capacity and grow geometrically. All callers provide actual
   required positions, and graph capture pre-grows tables before capture.

6. **NaN/Inf parity false positives**

   CLI and local L0 parity checks reject non-finite values and non-finite
   differences before reducing maximum absolute difference.

7. **Duplicate hash routes**

   Router validation rejects duplicate expert IDs in the selected hash prefix.

8. **Non-compressed graph warmup cursor**

   Host KV-validity metadata advances only after successful device KV write, so
   graph warmup and graph top-k use the same logical cursor.

9. **Shared-expert batched/decode dispatch drift**

   Enabling FP8 MMA initially made batched shared-SwiGLU use the rows kernel while
   token-loop/decode retained a scalar private matvec. The approximately
   `9.65e-6` layer-0 difference amplified across layers. The single-token path now
   calls the same rows implementation with `rows = 1`; 43-layer parity returned to
   elementwise `0.0` without widening tolerance.

## E1 execution ABI checkpoint

E1 is complete. This checkpoint changed the execution boundary and compatibility
failure handling without changing DSV4 mathematics or CUDA kernels.

Implemented boundary:

- the only public `ExecutionBatch` lives in `ferrule-common/src/execution.rs` and
  uses opaque `StateSlot`, `KvWriteSlot`, and `KvBlockId` identifiers, packed input
  vectors, ragged sequence spans, and explicit `ForwardMode`/`ForwardPhase`;
- top-k and full-logits requests have strict, distinct input/output semantics;
  phase/capability/query, position, block-table, KV write-slot, and output payload
  validation runs before or at the appropriate execution boundary;
- `PreparedModelPlan` and `ModelBatchExecutor` traits define neutral plan,
  validation, sequence lifecycle, and execute contracts;
- runtime-private `ScheduledBatch` owns request/session/KV correlation, while model
  output exposes only `input_row`; prefill action tokens move into the batch with
  `mem::take` instead of cloning the `Vec`;
- `TopKCompatibilityExecutor` is retained as a truthful single-sequence
  compatibility executor over `TopKModelRunner`; it is not a native multi-session
  `ModelBatchExecutor`. DSV4 CUDA advertises `max_top_k = 40`, which is validated
  before model mutation;
- the old graph-specific execution ABI is deleted. Reference graph execution uses
  an explicit `ReferenceGraphSequenceState` slice and returns low-level
  `TensorData`, not `ModelBatchExecutor`'s `ExecutionOutput`;
- prototype scheduler/preemption paths, duplicate graph entrypoints, implicit
  reference-graph default state, and obsolete aliases are deleted. Active runtime
  lowering uses `scheduling::actions`, private `ScheduledBatch`, and
  `graph::external_bindings`;
- model/output/commit/callback failures enter fail/poison handling. Failed metadata
  KV release keeps the handle in the failed sequence ledger, and a poisoned
  compatibility executor cannot yield its runner.

## Performance baseline

### Resolved 4096-token incident

Before this cut, one 4096-token layer took approximately `26.34s` and a full
single-layer prefill took `27.25s`. A 43-layer run exceeded six minutes before it
was stopped; linear projection estimated approximately 19 minutes.

The latest comparable measurements are:

| GB10 single-sequence measurement | Before | Current |
|---|---:|---:|
| `1L × 4096` measured layer body | `26.34s` | approximately `1.66s` |
| `1L × 4096` full prefill | `27.25s` | approximately `2.72s` |
| `5L × 4096` full prefill | not recorded | `13.14s` |
| `43L × 4096` prefill | did not complete | `118.651s` |
| 43L boundary test including decode | did not complete | `124.16s` |

The measured layer body is about `15.9×` faster. Full-model prompt processing is
approximately `34.5 tok/s`. The 43-layer number is an observed completion, whereas
the old 19-minute value was only a projection and must not be presented as a
measured speedup denominator.

| Stage/path | Previous | Current sync-profile value |
|---|---:|---:|
| FP8 `q_a` | approximately `490ms` | approximately `25ms` |
| FP8 `q_b` | approximately `3.97s` | approximately `215ms` |
| grouped BF16 `wo_a` / `output_a` | approximately `4.80s` | approximately `188ms` |

The production dispatch now uses:

- FP8 E4M3/E8M0 activation packing plus GB10 `m16n8k32` MMA for artifact linears;
- the model's official BF16 `wo_a` einsum semantics implemented as grouped

- one warp per `(token, head)` for the DSV4 512-wide sparse-attention case;
- one full-layer activation quantization, expert-major fixed-eight-column MoE
  segments, route-major output, and deterministic rank-order reduction;
- the same rows kernel for batched shared expert and rows=1 decode/shared expert.

### New bottleneck attribution

The scalar projection/attention/tile-MoE crisis is no longer the first-order
problem. Compressed layers still spend roughly `2–3.5s` in the combined
compressor/indexer/top-k path, and the full model still pays approximately 256 MB
in each direction at many layer boundaries for HC state. Expert transfer,
segment/route metadata uploads, and hundreds of thousands of temporary allocations
remain material.

The next performance work should therefore follow the ownership roadmap:

1. keep HC/hidden ping-pong on device across layers;
2. make compressor/indexer values and ring metadata device-resident;
3. move FP8 rows scratch, MoE segment workspace, and route output into a persistent
   arena;
4. reduce expert/metadata transfer through prepared handles and later stable device
   residency;
5. only then pursue reusable graphs and broader fusion.

The older cold five-token measurements (`17.47–17.75s` prefill,
`0.80–0.824 tok/s` decode, `18.76–19.01s` first token) predate this kernel cut.
They remain historical correctness-run data and must be rerun before use as current
headline throughput.

Do not use `FERRULE_DSV4_PROFILE_SYNC=1` or `FERRULE_CUDA_MOE_TIMING=1` for headline
throughput. They insert stream synchronization for attribution; asynchronous stage
timings can also shift work across stage boundaries, so use wall time for headline
results and synchronized profiles only for diagnosis.

## Current execution reality

The following statements are intentionally explicit:

- `DeepSeekV4Runner` still represents one implicit physical sequence.
- Runtime `PagedSequenceKvCache` is metadata/lifecycle only for this path.
- Physical CUDA window/combined/indexer KV remains model-runner/backend owned and
  layer-keyed.
- `ResidentScheduler` can describe batches, but the active executor rejects
  multi-row decode and mixed batches.
- Per-layer CUDA prefill exists, but HC rows return through host memory between
  layers.
- Appended prefill can fall back to token-wise decode execution.
- GPU compressor numerics are correct, but CPU shadow bookkeeping still adds
  D2H/H2D and is not retry-transactional.
- Router selection/residency/grouping remains host controlled in the active path.
- CUDA graph capture is segmented, token/position/route specific, and one-shot.
- Shared FFN still lacks a strict persistent allocation-free graph workspace.

These are the reasons performance remains far below vLLM/SGLang-class serving.
They are architecture gaps, not evidence that the corrected CUDA math should be
replaced.

## E0 closed and deferred gates

E0's architecture-migration gate is closed. The canonical 43-layer oracle is
bit-exact, scalar/F32/tile fallback oracles remain available, the 4096-token
continuation completes, short-window eager/graph parity passes, resource counters
attribute the path, and the prior six-minute non-completion incident has been
replaced by the completed `118.651s` prefill measurement. E1 is complete, and E2
is now the active implementation phase.

The remaining work is deferred to the phase that owns each lifecycle:

| Deferred gate | Owning phase | Scope |
|---|---|---|
| Allocation, arena-growth/D2D-preservation, and prepared-expert load/upload failure injection | E2 — prepared resources and arenas | Failed prepare/grow operations publish no partial resource and preserve the previous valid generation. |
| Generalized step commit, poison, reset, and compressor/indexer transition semantics | E3 — `SequenceExecutionState` | Extends the E1 single-runner compatibility safety contract to explicit per-sequence state; failed steps do not expose partially committed state. |
| Physical KV reserve → execute → commit/rollback | E5 — paged multi-plane KV | Reservations become visible only on commit; cancellation/failure rolls back pages, lengths, and refcounts. |
| Reproducible cold/warm TTFT and ITL p50/p95 plus throughput/counters | Continuous E1–E8 benchmark gate | Record phase-entry/exit regressions with profiling synchronizations disabled; this is not an E0 prerequisite. |

The post-MMA short-prompt rerun belongs to that continuous benchmark stream; the
existing pre-cut short-prompt numbers remain historical until it is recorded. They
did not block E1 and do not block active E2 work. Arena and long-lived
executable-cache hit counters become populated E2 and E7 gates when those resources
exist. Sequence interleaving is an E3 gate after explicit per-sequence state exists.
