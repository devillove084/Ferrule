# Ferrule Roadmap

_Last updated: 2026-07-11_

Ferrule is a Rust-native, state-aware LLM runtime for edge inference. The current
engineering center is **interactive, resident, CUDA-backed DeepSeek-V4 Flash /
DSpark execution** on GB10-class unified-memory machines, while **OLMoE** remains
the correctness golden model.

The working target is no longer only an offline `decode_tok/s` number. Ferrule
must optimize the user-visible path:

1. `./target/release/ferrule chat ...` reaches the prompt quickly;
2. every user turn appends new prompt tokens without redoing old prefix work;
3. the first response token and the full response arrive with minimal latency;
4. benchmark JSON still reports load / prefill / decode counters for regression
   tracking.

DSV4 drives missing runtime capabilities but must not become a model-specific
runtime architecture. Model-family names and raw HF tensor names stay in
`ferrule-model`; `ferrule-runtime` sees semantic roles, graph ops, artifact
groups, expert registries, KV state, residency policy, and backend object stores.

## Current capabilities

### Working now

- **OLMoE correctness fixture**: safetensors loading, GPU Q4_0 inference through
  cuda-oxide kernels, and router correctness around `norm_topk_prob`. CPU
  reference components remain the anchor for artifact decoders, router math, HC
  math, expert execution, and CUDA tests.
- **DeepSeek V4 / DSpark metadata and local artifacts**:
  - local model: `models/DeepSeek-V4-Flash-DSpark`;
  - `hidden_size=4096`, `moe_intermediate_size=2048`, `num_hidden_layers=43`;
  - `n_routed_experts=256`, `num_experts_per_tok=6`, routed expert dtype `fp4`;
  - DSpark metadata is parsed: `dspark_block_size=5`, noise token, target layers
    `[40, 41, 42]`, Markov rank `256`.
- **Full 43-layer DSV4 CUDA greedy path**:
  - `deepseek-v4-generate --backend cuda --json` reports load / prefill / decode
    timing and CUDA counters;
  - `ferrule chat ... -q cuda --chat-template deepseek-v4 --temp 0` runs the
    DSV4 greedy top-k fast path;
  - `lm_head` top-k can stay device-side for greedy decode.
- **Interactive DSV4 chat improvements (2026-07-07)**:
  - chat REPL becomes ready immediately while DSV4 artifacts load on a background
    thread;
  - CUDA context/operator cache is still created on the main thread;
  - prompt append uses `prefill_tokens_topk_interactive`: all prompt tokens except
    the last update KV/MoE/session state without materializing final hidden/logits;
    the last prompt token materializes top-k for generation;
  - `feed_token` now advances session state without unnecessary `hc_head`, output
    norm, or `lm_head` work.
- **DSV4 CUDA execution state**:
  - device-resident decode chain exists for the hot single-token path;
  - attention/KV append, norm weight cache, GPU output-head/top-k, batched routed
    MoE workspace, managed-memory FP4 expert handles, and mmap shard reads are in
    the working path;
  - `FERRULE_CUDA_MOE_TC=1` (default) enables the FP4 `mxf4` Tensor Core path;
    `FERRULE_CUDA_MOE_TC=0` remains a legacy decode-only diagnostic and currently
    rejects batched prefill, so it is not a complete scalar A/B fallback.
- **Blackwell FP4 Tensor Core smoke**:
  - local `fp4_mxf4_smoke` proves
    `mma.sync.aligned.m16n8k64.row.col.kind::mxf4.block_scale...ue8m0` on
    `sm_121a` with correct full-tile scale selection;
  - the integrated MoE path uses packed FP4 activations + E8M0 scales and has a
    fused `SwiGLU -> packed FP4` stage, eliminating the previous hidden-pack pass.
- **Expert residency / storage**:
  - `ExpertStreamingPlanner` tracks per-layer hot expert frequency and can
    prefetch routing-aware hot experts instead of naive low expert IDs;
  - bounded hotset residency (`--moe-hotset-experts`) exists for constrained
    machines but is not the fastest short-run default;
  - `HostStagedExpertCache`, mmap shard cache, and managed expert allocations are
    wired into CUDA DSV4.
- **Runtime graph foundation**: `ferrule-runtime::graph` opaque IR,
  `GraphProgram`, `ExecutionBatch`, `ExecutionOutput`, `ExternalBindingPlan`,
  `BackendObjectStore`, `ReferenceGraphBackend`, and shape validation registry.
- **Resident scheduling spine**: `ResidentScheduler`, `SequenceState`,
  `SchedulerAction`, `ResidentActionExecutor`, `ResidentTopKDriver`, and
  `PagedSequenceKvCache` now form the generic request → prefill/decode action →
  execution output → token streaming → finish/free-KV loop for tests and future
  serving.
- **DSV4 runtime-driver proof (2026-07-08)**: local full 43-layer DSV4 Flash / DSpark
  runs through `bench-interactive --runtime-driver` with `runtime_path =
  "resident_topk_driver"`. The runner remains in `ferrule-model`; runtime sees only
  `TopKModelRunner` plus scheduler/KV/action vocabulary. `PagedSequenceKvCache` is
  metadata/lifecycle only for this path; physical CUDA KV still lives in the DSV4 runner.
- **Model-family boundary**: `ferrule-model/src/families/` owns per-family HF
  tensor parsing. DSV4 raw tensor names are isolated in `families/deepseek_v4.rs`.
- **Storage/residency vocabulary** (`ferrule-runtime::storage`):
  `StorageObjectId`, `ObjectLocator`, `Placement`, `ObjectReplica`,
  `ReplicaHandleId`, `StorageCatalog`, `TransferEngine`, and
  frequency-aware scoring.
- **Quantization / artifacts**: Q4_0/Q8_0 full path, mixed precision policy,
  FP4 E2M1 + E8M0 artifact decoders, FP8 E4M3FN decoders, KV quantization, and
  WeightPack reader/writer for OLMoE.
- **CLI surface**: `info`, `cuda`, `chat`, `bench-interactive`,
  `inspect-weightpack`, `expert-stream-smoke`, `deepseek-v4-probe`, and
  `deepseek-v4-generate`. Serving remains a runtime design target, not an
  active CLI command in the current workspace.

### Main gaps

Current measurements are on GB10 / `sm_121a` with the local
`models/DeepSeek-V4-Flash-DSpark` artifact unless noted. Batched prefill vs
single-token append is now bit-exact through all 43 layers. The main gap has moved
from an unidentified numerical bug to **execution ownership and batching**: the
scheduler can describe batches, but the production CUDA runner, physical KV, expert
residency, and graph lifecycle are still effectively single-session and model-owned.

| Area | Current known state | Gap |
|---|---|---|
| Correctness | deterministic route-ranked MoE reduction, GPU compressor parity, and missing non-compressed KV QAT fixes produce three independent 43L traces with every layer `max_abs_diff=0.0`; latest cut top-1 is `30594` | add full-model >4096, graph eager/replay, failure/reset, and multi-session interleaving regressions without changing the established floating-point fold order |
| Long-context safety | typed RoPE tables validate dimension/YaRN parameters, retain a 4096 initial capacity, and grow geometrically; GB10 kernel coverage passes positions 4095/4096 | run a real long-context model continuation and keep all graph buckets pre-grown before capture |
| Chat startup | REPL prints immediately; local artifact load is about one second in the latest cold run | CUDA/operator and expert-residency warmup still happens on the first real prompt |
| Runtime-driver full path | three independent cold 43L `Hello`, `n=2` runs emitted `[30594, 1175]`; cold prefill is `17.47–17.52s`, decode `0.80–0.82 tok/s` | cold expert materialization and host-controlled execution dominate; historical preserved-residency runs remain much faster |
| Prompt/prefill | per-layer CUDA rows/tile execution and deterministic batched MoE exist | the runner still downloads the HC result at layer boundaries and re-uploads it for the next layer; appended chunks can still fall back to token-wise decode |
| MoE prefill | grouped expert execution plus per-expert scratch and route-ranked device reduction are bit-exact with decode | move routing/grouping metadata and expert indirection fully device-side, then improve packing across real batch rows |
| Compressor/indexer | GPU softmax/RMS/RoPE/QAT now determines compressed values and matches token-loop numerics | CPU shadow state and D2H/H2D bookkeeping remain in the hot path and are not retry-transactional |
| Expert residency | host-staged/pinned/in-flight CUDA mechanisms exist | planner/policy lives inside the model runner, selected misses remain host-controlled, and there is no cross-request residency coordinator |
| CUDA graph | segmented capture/replay scaffolding and persistent attention/MoE arenas exist | current graph is token/route specific and one-shot; shared FFN and some temporaries are not pointer-stable, so capture is not a steady-state execution system |
| Serving spine | scheduler/actions can describe multiple sequences and runtime owns metadata KV handles | executor rejects multi-row decode/mixed batches; physical DSV4 CUDA KV is a separate single-session ledger not addressed by runtime handles |
| DSpark | metadata parsed | no real speculative block decode, verification loop, or rollback-safe KV/residency state |

Important measurement corrections:

- Short `run:` wall-clock timings are misleading; use JSON summary for decode
  throughput and explicit chat/`bench-interactive` stats for interactive latency.
- `FERRULE_DSV4_PROFILE_SYNC=1` and `FERRULE_CUDA_MOE_TIMING=1` insert stream
  synchronization and are for attribution, not headline throughput or a
  correctness fix.
- The current 43-layer bottleneck is not runtime scheduling: the runtime-driver
  path uses one prefill action followed by decode actions.
- CUDA graph replay cannot pay off while host routing, expert streaming, and
  compressor/top-k boundaries remain in the hot path.
- Bounded hotset residency can reduce resident handles, but on short runs it can
  increase churn and slow down throughput.
- Duplicating the same hidden vector to fake `dspark_block_size=5` would be wrong;
  real GEMM utilization needs real per-column hidden states and route semantics.

## Architecture principles

1. **Keep CPU reference components alive** — artifact decoders, router math,
   HC math, and expert executors are correctness anchors. The legacy full-model
   CPU forward path has been removed; reference is component-level, not a runner.
2. **Model families are policies, not runners** — plug into layout/attention/
   router/quant/residency policies; never fork a runner per family.
3. **No privileged target-model path** — generic crates expose semantic roles
   and policy traits, not DeepSeek-named fields.
4. **Measure before optimizing** — add benchmarks and counters before kernel
   rewrites.
5. **Treat WeightPack as a runtime artifact** — Ferrule's local/edge deployment
   unit with manifest, chunks, checksums, and future placement metadata.
6. **Preserve MoE semantics** — router logits, top-k, expert weights, and
   normalization are correctness-critical and must stay explicit.
7. **Compose behavior through typed policies** — `EnginePlan`/config surface,
   not `if model_name` branches.
8. **Make unsupported execution explicit** — `info` may inspect metadata;
   executable entrypoints such as `chat` or future serving must fail with a named
   missing-policy error.
9. **Async I/O is allowed for expert loading** — the correctness path uses
   synchronous bounded reader; expert weight loading uses tokio + `pread` for
   concurrent disk reads. Planning semantics remain synchronous and unchanged.
10. **PK against real engines with reproducible commands** — every perf claim
    needs prompt, context, quant, batch, hardware, and quality note.

## Composable engine architecture

A new model bring-up provides descriptor parsing, tensor layout, and policy
implementations; the runtime composes them into an `EnginePlan`:

```
ModelDescriptor → ModelLayout → ModelSupportContract → EnginePlan {
  ModelFamilyPolicy, AttentionPolicy, RouterPolicy, ExpertPolicy,
  QuantPolicy, KvPolicy, SchedulerPolicy, ParallelismPlan,
  ResidencyPolicy, SpeculationPolicy
}
```

| Policy | Responsibility | Near-term default | Future switches |
|---|---|---|---|
| `ModelFamilyPolicy` | map metadata/tensor names → semantic layer components | OLMoE fixture, DSV4 descriptor | family plugins without runner forks |
| `AttentionPolicy` | MHA/GQA/DeepSeek attention, RoPE, cache layout | GQA RoPE | CSA/HCA/MLA, Flash prefill |
| `RouterPolicy` | logits, bias, hash routing, top-k normalization | dense top-k | bias/hash routing, EP routing |
| `ExpertPolicy` | dense MLP, routed/shared experts, activation | routed experts | shared experts, expert batching, EP |
| `QuantPolicy` | tensor quant class, dequant/matvec, conversion | Q4_0/Q8_0 WeightPack | GGUF K/IQ, FP4/FP8, mixed |
| `KvPolicy` | contiguous/paged/quantized KV ownership | contiguous per-session | paged KV, prefix/radix reuse |
| `SchedulerPolicy` | prefill/decode admission, chunking, batching | single request | continuous batching, preemption |
| `ParallelismPlan` | DP/TP/EP/SP/CP/PP placement | single process/GPU | all combinations |
| `ResidencyPolicy` (model-level) | all-GPU vs streaming allowed | all resident if it fits | expert/layer streaming |
| `StorageResidencyPolicy` (runtime-level) | per-tier budgets, eviction, prefetch | — | frequency-aware, host-staged |
| `SpeculationPolicy` | none/MTP/draft-model proposal + acceptance | none | MTP, DSpark, Eagle |

**Two residency policy levels** (see `docs/storage-residency-architecture.md`):

- `ferrule-model`'s `ResidencyPolicy` (`streaming_allowed`,
  `all_resident_required`) — model-level: does this model fit in GPU memory?
- `ferrule-runtime::storage`'s `StorageResidencyPolicy` (`Budgets`,
  `EvictionWeights`, `prefetch_window`) — runtime-level: how much to keep on
  each tier, what to evict. Active only when `streaming_allowed = true`.

### Execution ownership target toward vLLM/SGLang-class serving (2026-07-11 audit)

The next performance phase is an ownership refactor, not a broad kernel rewrite.
The current code has useful mechanisms, but their lifecycle is composed in the
wrong place:

- `DeepSeekV4Runner` owns model-global resources, one implicit sequence,
  physical KV, expert prediction/residency, CUDA graph capture, and telemetry;
- `DeepSeekV4OperatorContext` plus `DeepSeekV4CudaOperatorCache` form a large
  service locator/backend with string-keyed resources and many eager/device/graph
  variants;
- `ResidentScheduler` can produce multi-row actions, while
  `ResidentActionExecutor` still rejects multi-session prefill, mixed batches, and
  decode batches larger than one;
- `PagedSequenceKvCache` is a runtime metadata ledger, while the kernels consume a
  separate layer-keyed DSV4 CUDA KV ledger.

This is the core difference from the stable scheduler → KV manager → model runner
contract in vLLM and the scheduler/forward-mode/memory-pool split in SGLang. Ferrule
should copy the ownership principles, not their Python/plugin surface.

```text
GenerateRequest / Session
        │
        ▼
ferrule-runtime
  admission + scheduling + sampling
  sequence handles + physical page ownership
  expert-residency coordinator + graph-bucket lifecycle
        │ ModelBatch / StateBinding / LogitsRequest
        ▼
prepared model executor
  immutable model plan + typed device handles
  persistent per-{phase, shape bucket} arenas
  prefill/decode lowering over explicit sequence rows
        │ stable buffers, page tables, expert indirection
        ▼
ferrule-cuda
  kernels + streams/events + physical device resources
```

| Layer | Must own | Must not own |
|---|---|---|
| `ferrule-model` | validated config, typed weights/layout, model semantics, CPU/component reference, model-specific KV schema and lowering policy | request queues, one global session position, CUDA graph lifecycle, host/device transfer policy |
| `ferrule-runtime` | request/session separation, continuous-batch planning, sampling/stop state, sequence/KV page lifecycle, preemption, cross-request expert policy, executable/graph cache lifecycle | DSV4 tensor names, compressor math, concrete CUDA buffers |
| prepared backend executor | model-global prepared handles, per-sequence bindings, persistent arenas, flattened/ragged batch metadata, eager/graph execution through the same `*_into` pipeline | service-level request semantics or artifact discovery in the token hot path |
| `ferrule-cuda` | allocations, streams/events, typed resource tables, multi-plane physical KV, expert handles/indirection, kernels | scheduler fairness, request/session identity, model-family policy decisions |

The neutral execution contract must carry, at minimum:

- explicit sequence/state handles for every row;
- packed token IDs, positions, sequence lengths, query offsets, and KV slot/page
  mappings;
- `Prefill` / `Decode` / `Mixed` forward mode and ragged row metadata;
- per-row `LogitsRequest::{None, TopK, Full, Sample}` intent;
- backend capabilities and shape/bucket constraints.

Do not replace the current service locator with one enormous generic trait. Start
with a small batch executor boundary, typed prepared handles, and backend-owned
arenas. DSV4-specific HC/compressor/indexer semantics remain model policies; their
request and resource lifecycle moves upward/downward to the appropriate owner.

Migration invariants:

1. Keep CPU/component reference, token-loop prefill, layer/attention checkpoints,
   deterministic ranked MoE reduction, and compressor parity as differential
   oracles until the new executor is bit-exact.
2. Split model-global resources from per-sequence state before enabling more than
   one active request.
3. Make one physical KV source of truth before prefix reuse, preemption, DSpark
   rollback, or P/D transfer.
4. Eager and graph execution must consume the same prepared buffers and kernels;
   graph capture is a cache of stable execution, not a third implementation.
5. Hot-path acceptance criteria are zero device allocation, zero host-dependent
   intermediate D2H, no stream-wide synchronization, one compact metadata update,
   and graph reuse across many iterations for a fixed bucket.

## Priority roadmap

The priority order is now based on the user-visible interactive path:

```text
startup → prompt append/prefill → first token → steady decode → serving/multi-session
```

### P0 - Correctness and observability gates

- [x] JSON benchmark summary for `deepseek-v4-generate --json` with decode-only
      counters after prefill/warmup baseline subtraction.
- [x] MoE timing counters behind `FERRULE_CUDA_MOE_TIMING=1`.
- [x] DSV4 chat `stats>` line reports per-turn prefill/decode wall time.
- [x] Add a dedicated interactive benchmark command (`bench-interactive`) that
      feeds multiple turns and reports time-to-REPL, artifact load,
      first-token latency, per-turn prefill, per-turn decode, generated tok/s,
      and resident/cache counters.
- [x] Add golden interactive traces (`configs/golden/`) with token IDs and stop
      behavior for a fixed two-turn prompt under greedy decoding;
      `bench-interactive --golden <path>` compares live output against golden.
- [x] Add DSV4 model-internal attribution to `bench-interactive --json`: operator
      counters, per-layer bind/state/prefill/decode/attention/MoE timing,
      attention-stage breakdown, output-head detail, and profiling sync controls
      (`FERRULE_DSV4_PROFILE_SYNC=1`, `FERRULE_CUDA_MOE_TIMING=1`).
- [x] Prefill parity harness (`deepseek-v4-prefill-parity`): compares batched

- [x] Optional selected-layer parity checkpoints at `attention_input`,

      combined-KV growth test that verifies device values survive capacity growth.
- [x] Make batched/decode MoE accumulation deterministic without host stream

      reproduces the token-loop left fold, including the shared-FFN starting value.
- [x] Move CUDA compressor output numerics onto the same GPU softmax/RMS/RoPE/QAT
      chain as batched prefill; three 5L and three 43L runs are elementwise `0.0`.
- [x] Harden cache/resource contracts: typed growable RoPE metadata with 4095/4096
      CUDA coverage, failure-atomic combined-KV growth, finite-value L0 assertions,
      and hash-route duplicate rejection at router validation.
- [x] Advance non-compressed host KV validity metadata only after successful device
      writes so graph warmup state and graph top-k masking use the same cursor.
- [ ] Add a real-model >4096 prompt/continuation regression and a prompt shorter
      than `window_size` eager-vs-graph replay regression.
- [ ] Define and test request-fatal reset semantics for compressor/attention errors;
      retryable execution requires explicit stage/commit or rollback.

### P1 - Resident interactive engine

Goal: match SGLang-style execution at the architecture level before chasing more
micro-kernels: a long-lived engine owns model artifacts, CUDA context, KV/cache,
residency state, and request/session state.

- [x] DSV4 chat REPL starts immediately while artifacts load in the background.
- [x] CUDA context stays on the main thread; background loader only materializes
      CPU-side model artifacts.
- [x] `feed_token` appends session state without materializing final hidden/logits.
- [x] `TopKModelRunner::prefill_tokens` gives runtime a no-logits prompt append
      capability, so non-final prefill chunks no longer have to materialize top-k.
- [x] `prefill_tokens_topk_interactive` now routes through the DSV4 batched/segment
      prefill core and materializes top-k only for the final prompt chunk/token.
- [x] Replace the CLI-local lazy runner loop with a reusable runtime
      `EngineWorker`/`LazyEngineWorker` abstraction over `TopKModelRunner`;
      DSV4 chat and `bench-interactive` now execute turns through it.
- [x] Split worker execution from single-turn `generate_turn` into explicit
      `append_prompt` + `decode_next` phases with a concrete `TopKDecodeState`.
- [x] Add structured session finish reasons (`SequenceFinishReason`, re-exported
      as `TopKFinishReason` for top-k turns) and minimal `cancel_decode` on top
      of the phased API; rollback/KV rewind remains future work.
- [x] Expose structured worker stats (`EngineWorkerStats`) for position, tracked
      tokens, turns, prompt/generated totals, and bound layer count.
- [x] Connect session/scheduler/graph vocabulary: `SequenceState` now tracks
      request id, request sampling/stop/budget, KV handle, logical position, and
      prefill cursor; `SchedulerAction` materializes prefill/decode work into
      `ExecutionBatch`, paired with `ExecutionOutput` for backend results.
- [x] Add `ResidentScheduler` + `ResidentActionExecutor` + `ResidentTopKDriver`
      bridge so the runtime can drive `GenerateRequest -> SchedulerAction ->
      ExecutionBatch -> TopKModelRunner -> ExecutionOutput -> token event ->
      SequenceState/KV finish` without owning a concrete model.
- [x] Prove the bridge with real local full 43-layer DSV4 via
      `bench-interactive --runtime-driver` / `just dsv4-runtime-driver-bench`.
- [x] Add `PagedSequenceKvCache` so paged block tables implement `SequenceKvCache`
      admission/free lifecycle. CUDA paged-attention execution remains future work.
- [x] Avoid unnecessary next-logits work on the final max-token decode step in the
      scheduler/action executor path via `DecodeAction::require_logits`.
- [x] Report real prompt-to-first-token latency (`ttft`), prefill tok/s, decode
      tok/s, final position, finish reason, optional warmup timing/counter
      baselines, `runtime_path`, runtime-driver action stats, and DSV4 prefill path
      stats in chat/interactive benchmark output.
- [ ] Introduce a small neutral batched model-executor contract carrying explicit
      sequence handles, positions, ragged offsets, state/KV bindings, forward mode,
      and per-row logits intent. Keep the current `TopKModelRunner` as a temporary
      single-sequence adapter.
- [ ] Split DSV4 model-global prepared resources from per-sequence attention,
      compressor, position, and graph-input state; remove the runner-global session
      as the physical execution identity.
- [ ] Move request/session/sampling and graph-bucket lifecycle to runtime; move typed
      device resources and persistent arenas to the backend. Artifact binding remains
      model-owned, while expert residency policy becomes runtime-coordinated.
- [ ] Persistent warmup policy: after artifact load, optionally warm common CUDA
      kernels/linears without blocking the REPL.

### P2 — True device-resident CUDA prefill

Goal: eliminate host `Vec<f32>` boundaries and token-level full-layer loops during
prompt append. This is the current largest interactive latency gap and the first
feature to land before kernel tuning.

- [x] First correctness-first DSV4 segment prefill vertical slice:
      `tokens_per_chunk > 1`, existing session prefix, per-layer processing through
      the runner-owned DSV4 state.
- [x] Attention segment append updates window KV, compressed KV, and indexer state
      without resetting compressor state for appended chunks; prompt-start chunks
      still use the start-prefill compressor path.
- [x] Only materialize top-k/logits for final prefill chunks in the runtime driver;
      non-final chunks call `TopKModelRunner::prefill_tokens`.
- [x] Add chunk-size and max-layer controls to `bench-interactive`:
      `--prefill-chunk-size` and `--max-layers`, with per-turn runtime/DSV4 prefill
      stats in JSON.
- [x] Add per-layer DSV4 CUDA batched prefill for HC, attention, grouped MoE, and HC
      post, with final top-k materialized only when requested.
- [ ] Keep prompt chunk hidden/HC states device-resident across *all* layers; the
      runner currently downloads each layer result and re-uploads it for the next
      layer, so `Vec<f32>` is still the outer prefill contract.
- [ ] Replace appended-segment token-wise decode fallback with true ragged/chunked
      CUDA prefill while preserving compressor/window/indexer continuation state.
- [ ] Make parity/debug the only path allowed to download intermediate hidden rows.
- [ ] Add automated chunk-size sweep benchmark for chat prompts and full 43-layer
      `Hello` profiles, tracking launches, H2D/D2H copies, expert loads, and stage
      attribution.

### P3 — Device-resident compressor/indexer path

Goal: remove remaining host boundaries that block efficient chunked prefill and
CUDA graph capture.

- [x] Device-resident decode chain exists for the hot single-token path.
- [x] Norm weights, typed growable RoPE tables, output-head top-k, and MoE workspace
      are cached.
- [x] GPU softmax/RMS/RoPE/QAT determines compressor output values and is bit-exact
      with batched prefill.
- [x] CUDA indexer query/top-k and persistent indexer/combined KV paths exist.
- [x] Combined window/compressed values survive device-side capacity growth.
- [x] CPU/token-loop and CUDA paths have layer plus attention-stage parity hooks.
- [ ] Remove compressor CPU shadow D2H/H2D bookkeeping by storing ring state and
      metadata on device; define transactional failure/reset behavior.
- [ ] Lower compressor/indexer state through explicit per-sequence bindings so
      multiple sequences can share one prepared executor safely.

### P4 — Expert residency and batched FP4 MoE execution

Goal: reduce MoE cost by changing execution shape before micro-optimizing kernels.
Do not fake DSpark batching; use real hidden columns and real per-column routes.

- [x] Managed-memory FP4 expert handles default-on for GB10 unified memory.
- [x] Host mmap cache + `HostStagedExpertCache` are wired.
- [x] Routing-aware hot expert prediction replaces naive low-ID prefetch.
- [x] Batched selected-expert MoE workspace avoids per-call scratch allocation.
- [x] FP4 `mxf4` Tensor Core path is integrated and default-on. The legacy
      `FERRULE_CUDA_MOE_TC=0` diagnostic does not cover batched prefill and must not

## Implementation order

1. **Freeze the correctness oracle — done for the current cut.** Preserve the
   three-run 5L/43L bit-exact traces, `[30594, 1175]` continuation, component CPU
   reference, deterministic reducer, compressor checkpoints, and non-finite guards.
2. **Close state-safety gates.** Finish real-model >4096, graph eager/replay, cache
   growth, and request-fatal reset tests before changing execution ownership.
3. **Introduce the neutral batch boundary.** Add explicit sequence/state handles,
   ragged positions/offsets, KV bindings, forward mode, and logits intent. Adapt the
   existing single-sequence `TopKModelRunner` without changing outputs.
4. **Split ownership.** Separate immutable model-global prepared resources from
   per-sequence attention/compressor state; move request/sampling/graph lifecycle to
   runtime and device resources/arenas to the backend.
5. **Build an immutable prepared CUDA plan and persistent arenas.** Replace hot-path
   artifact discovery, strings/HashMaps, and returning-buffer APIs with typed handles
   and `*_into` execution shared by eager and graph modes.
6. **Finish device residency.** Keep prefill HC/hidden across all layers, move
   compressor/indexer ring state and router/grouping metadata fully device-side, and
   eliminate intermediate D2H/H2D except explicit diagnostics/final token output.
7. **Implement real multi-session physical KV.** Bind runtime sequence handles to
   DSV4 multi-plane window/compressed/indexer page tables; then enable multi-row
   decode, ragged chunked prefill, mixed batches, prefix reuse, and preemption.
8. **Move expert policy to a runtime coordinator.** Keep CUDA expert handles,
   pointer tables, workspaces, and events backend-owned; batch/group real tokens by
   expert and make rare misses event-driven instead of stream-wide synchronized.
9. **Capture stable piecewise graph buckets.** Patch compact device metadata and
   reuse graph execs across tokens/requests; do not retain the current one-shot graph
   as a separate semantic implementation.
10. **Then tune kernels and serving.** Profile WO-A, fusion, grouped FP4 packing, and
    output sampling after the execution shape is correct; compare with
    vLLM/SGLang/TRT-LLM only using reproducible model/quality/batch measurements.
    DSpark proposal/verify/rollback follows stable per-sequence state.

## What not to do

- Do not claim continuous batching because the scheduler can construct a
  `DecodeBatch`; require a multi-sequence executor and physical CUDA KV bindings.
- Do not implement radix cache before prefix cache and shareable physical KV pages exist.
- Do not perform a flag-day rewrite of `attention.rs` / `cuda_cache.rs` or change
  floating-point fold order while moving code. Extract ownership mechanically and
  require layer/attention parity after every slice.
- Do not delete CPU/component reference, token-loop prefill, parity checkpoints, or
  ignored real-model CUDA regressions — they are the current correctness anchors.
- Do not add broad model-family support before loader/runtime boundaries are
  cleaner.
- Do not rewrite all CUDA kernels before `bench-interactive` / JSON diagnostics
  and profiling exist.
- Do not jump to sparse-attention micro-optimization while layer-boundary transfers,
  host compressor/router state, and selected-expert materialization dominate.
- Do not optimize or extend token-specific one-shot CUDA graphs; first establish
  stable prepared arenas, device metadata patching, and a single eager/graph pipeline.
- Do not optimize `ResidentScheduler` for the current 43-layer `Hello` latency;
  runtime-driver stats show model execution, not scheduling, is the bottleneck.
- Do not add a new quantization format without a logits/parity, perplexity-style,
  or golden-token harness.
- Do not make WeightPack format changes without manifest/version checks.
- Do not add DeepSeek-specific fields to generic scheduler/sampler/session/
  executor — add a policy/layout role instead.
- Do not claim DSV4 execution until attention, router, experts, auxiliary
  tensors, and quant classes are all accounted for.
- Do not claim DSpark support until generic target/draft speculation with
  acceptance/rejection/rollback/metrics is implemented.
- Do not compare Ferrule against llama.cpp/vLLM/SGLang without recording model,
  quant, prompt, context, batch, hardware, and command.
- Do not block on async I/O — keep the synchronous bounded reader as the
  correctness path.
- Do not enable DSpark before base DSV4 correctness and device-resident decode
  are stable.
- Do not treat `PagedSequenceKvCache` as physical DSV4 CUDA KV; in the current
  runtime-driver path it is metadata/lifecycle while physical KV remains runner-owned.

## DSV4 synthetic fixture benchmark

This section is retained as historical context only. The current optimization
priority is based on the real local model and interactive chat measurements in
the GB10 section below.

### Fixture purpose

`scripts/dsv4_synthetic_fixture.py` creates structurally valid DSV4-like
safetensors without requiring the full 166 GB real model. It is useful for
loader, inventory, graph binding, expert streaming, and residency development.

The fixture preserves:

- DSV4-style top-level config and tokenizer boundary;
- layer counts and semantic tensor roles;
- FP8 block-aligned attention artifacts;
- FP4 routed expert packing and E8M0 scale shapes;
- shard/index structure compatible with the real loader.

### What the old fixture measurements proved

Historical synthetic/fixture work helped validate these architecture changes:

1. DSV4 artifact inventory and typed binders can materialize complete layer
   payloads.
2. Expert streaming can load bounded safetensors byte ranges and install resident
   handles.
3. Batched selected-expert MoE and device-resident attention pieces reduce obvious
   host/device boundaries.
4. JSON counters are required; raw wall-clock command timing was too noisy.

### What is now stale

The old fixture-era bottleneck attribution, next-step list, and git-state notes
were removed because they conflict with the real-model measurements. In
particular:

- CUDA graph capture is no longer the first priority while host routing, expert
  streaming, compressor/top-k, and prefill `Vec<f32>` boundaries remain.
- The current highest-priority user-visible bottleneck is the host-heavy DSV4
  prefill/model execution shape; this needs true device-resident prefill and
  batched layer-level MoE prefill.
- MoE Tensor Core work remains important, but the current `batch_cols=1` path is
  not enough for the 5 tok/s target or for honest DSpark utilization.

Implementation note: `ExpertResidencyPlan` is now model-agnostic in `moe::streaming`; CUDA MoE consumes it instead of mixing residency classification, host staging, selected materialization, and prefetch enqueue in one loop. Generic planning now calls this backend in-flight state `materializing`; the CUDA implementation maps it to `Uploading { event }`. Upload tickets own pinned host sources plus CUDA handles, record an event on a dedicated upload stream, and are finalized into the resident expert cache only after event completion. Completed prefetch uploads are now returned to the streaming planner via `commit_step_loaded` so planner residency does not lag backend residency. A CUDA pinned expert-source LRU (`FERRULE_DSV4_PINNED_EXPERT_CACHE_CAPACITY`, default 64) now backs async prefetch uploads, so upload tickets clone pinned Arc sources instead of re-pinning payload bytes. Runtime-driver warmup reset now has a precise state contract: preserve persistent weights/expert residency, but clear scheduler/session state, host KV state, CUDA window/combined/indexer KV caches, and graph/MoE scratch arenas. The earlier preserved-reset `[19923]` bug was stale device-side sequence cache contamination, not a model-routing change.

Correctness revalidation on the actual GB10 workspace (updated 2026-07-11): the local model is present at `models/DeepSeek-V4-Flash-DSpark`, and all results below use a CUDA 13.0 `sm_121a` release build produced by `cargo oxide`. The earlier `[19923]` probe files remain useful historical evidence, but the statement that the model was unavailable or that the fixes were not model-validated is obsolete.

#### Closed prefill parity, continuation, and long-context safety (2026-07-11)

Three independent numerical causes were fixed rather than masked with synchronization:

1. non-compressed CUDA decode now applies the same post-RoPE attention KV QAT as
   batched prefill;
2. routed MoE writes each expert result to persistent scratch and reduces it in
   router-rank order, starting from the shared-FFN output, instead of using relaxed
   atomics or a differently parenthesized host/device reduction;
3. incremental compressed decode now uses the same CUDA softmax, RMS, RoPE, and QAT
   chain as batched compressor prefill. Host shadow state advances control metadata
   but no longer determines compressed values.

| Probe | Result | Interpretation |
|---|---|---|
| CUDA 5L parity, three independent runs | every layer `max_abs_diff=0.0` | deterministic MoE and compressor paths close the former 1-ULP/amplification chain |
| CUDA 43L parity, three independent runs | `first_diverging_layer=null`; every layer `0.0`; final cut top-1 `30594` | full-depth batched prefill and token-loop append are bit-exact |
| Fresh 43L parity on 2026-07-11 | cuts 1/5/23/43 match at `83484/83484/108527/30594`; all 43 layers `0.0` | typed RoPE/cache safety changes preserve the closed oracle |
| Runtime-driver 43L, cold, `n=2`, three independent processes | `[30594, 1175]` in all three runs | prefill→decode continuation is stable for the five-token chat prompt |
| RoPE CUDA boundary regression | positions 4095 and 4096 match CPU YaRN reference after `4096→8192` growth | long-context table access no longer relies on a fixed 4096 allocation |
| Combined-KV CUDA growth regression | window and end-of-capacity compressed rows survive `16→32` growth | expansion preserves the authoritative device values |

```bash

  models/DeepSeek-V4-Flash-DSpark Hello 43 cuda --chat --json
```

The harness supports `FERRULE_DSV4_PARITY_CHECKPOINT_STAGE=all` (or `*`) and
captures both layer stages (`attention_input`, `after_attention`, `ffn_input`,
`moe_output`, `layer_output`) and attention stages (`attn_q_latent`, `attn_q_norm`,
`attn_query`, `attn_kv`, `attn_indexer_compressed`, `attn_main_compressed`,
`attn_context`, `attn_context_rope`, `attn_output_a`, `attn_output`). Selection is
cached when the operator context is built; disabled runs add no environment lookup,
D2H, or synchronization. CLI and local L0 comparisons reject non-finite inputs and
differences instead of letting `f32::max` hide NaN.

Long-context/cache hardening in the same cut:

- RoPE cache entries now bind `rope_dim`, complete `DeepSeekV4RopeParams`, and
  capacity to one typed resource name. Same-name parameter changes fail explicitly;
  capacity grows geometrically and replacement occurs only after both new device
  tables upload successfully.
- every prefill/decode/indexer/compressor caller supplies its actual required maximum
  position; strided compressor tables account for source position stride;
- graph-safe decode pre-grows main/indexer RoPE resources before stream capture and
  only validates stable table identity/capacity inside capture;
- combined-KV growth keeps the old map entry and capacity metadata installed until
  replacement allocation plus D2D copy succeeds;
- non-compressed device decode advances host validity metadata only after the device
  KV write succeeds, preventing warmup/capture top-k from using a stale window length;
- hash routing rejects duplicate experts in the selected prefix at the generic
  router contract rather than failing deep inside CUDA prefill.

Harness scope remains important: the parity command compares prompt-row hidden state
and resets between paths. It does not alone prove cache continuation, graph replay,
failure rollback, or multi-session isolation. Remaining P0 coverage is a real-model
prompt crossing 4096, eager-vs-graph replay with prompt length below the attention
window, request-fatal reset behavior, and two independently interleaved sequences.

The three-run cold correctness baseline is `prefill_s=17.47–17.52`, decode
`0.80–0.82 tok/s`, first token `18.76–18.83s`. A post-RoPE-safety revalidation
emitted the same `[30594, 1175]` at `17.75s`, `0.824 tok/s`, and `19.01s`; this is
normal cold residency variance, not a visible deterministic-reduction regression.
These are not SOTA numbers:
first-turn expert materialization, layer-boundary prefill transfers, CPU compressor
shadow bookkeeping, host routing/residency, and single-row execution remain dominant.

CUDA graph remains an experimental one-shot mechanism: replay enqueue is cheap, but
capture includes an eager warmup and the graph is tied to one token/position/route
shape. The next graph step is a prepared persistent arena plus patchable batch/KV/
route metadata, not more counters or another monolithic full-model capture.

### First batched-prefill MoE / hotset cut (2026-07-08)

Implementation cut:

- prefill layer MoE no longer calls `routed_moe_step` in a token loop;
- CUDA path routes the batch, plans residency by tile, groups tokens by selected
  expert, gathers rows into compact device batches, runs `batch_cols <= 8` FP4 TC
  expert execution, and scatter-adds results back to `[tokens, hidden]`;
- shared FFN uses batched artifact-linear row execution;
- CUDA expert handles are retained as an operator-cache hotset across warmup and

## Review checklist

For every patch:

- **Behavior**: does it change inference output? Is the change tested?
- **Correctness**: do CPU reference components still pass? Do logits/parity or
  golden-token diagnostics regress?
- **Generality**: does it add model-specific names to generic crates?
- **Performance**: does it add host sync to a device path? Does it add kernel
  launches?
- **State ownership**: does it leak state into the wrong crate?
- **Compatibility**: does it break existing CLI commands or WeightPack files?
- **Safety**: does it add `unsafe` without justification?
